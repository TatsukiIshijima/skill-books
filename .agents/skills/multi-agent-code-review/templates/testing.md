# テスト方針

## テスト種別と配置

| 種別 | 対象 | 配置パス | 実行方法 |
|---|---|---|---|
| (例) ユニットテスト | Domain / Application のロジック | `tests/unit/**` | `npm test` |
| (例) 統合テスト | API エンドポイント | `tests/integration/**` | `npm run test:integration` |

## 必須テストの基準

<!-- 「この変更にはテストが必須」という条件。
testing 観点は、これに該当する変更にテストが伴わない場合 Medium 以上で指摘する。 -->

- (例) Domain / Application 層の public メソッドの追加・挙動変更には
  ユニットテストの追加・更新が必須
- (例) バグ修正には、そのバグを再現するテストケースの追加が必須

## テスト不要な変更

<!-- テストを求めない変更を明示しておくと、無駄な指摘を防げる。 -->

- (例) UI レイアウトのみの変更
- (例) ログ出力・コメントのみの変更

## 境界値・異常系の基準

- (例) 数値パラメータを受けるロジックは 0・負数・最大値のケースを含める
- (例) 外部 API 呼び出しを含むユースケースは失敗系(タイムアウト・エラーレスポンス)を含める

## モック方針

- (例) 外部 API・DB はモックする。Domain 内のクラスはモックせず実体を使う
- (例) 時刻は `Clock` インターフェース経由で注入する(`DateTime.Now` 直接参照禁止)

## カバレッジ

<!-- 数値目標がある場合のみ。無理に決めなくてよい。 -->

- (例) Domain 層は行カバレッジ 80% 以上を維持する
