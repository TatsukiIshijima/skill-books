# PR Review Triage

PRの変更内容を分析し、人間によるレビューが必要かどうかを判定する。
判定は2段階のハイブリッド方式で行う: まず決定的なルールで明白なケースを即決し(Stage 1)、
どちらとも言えないグレーゾーンだけdiffを精読して評価する(Stage 2)。
明白なケースをルールで済ませるのは、判定を安定させ、同じPRに対して結果が揺れないようにするため。

## 前提条件

- GitHub CLI (`gh`) が認証済みであること。最初に `gh auth status` で確認し、未認証ならユーザーに案内して中断する。
- 対象PRのリポジトリ内(またはリポジトリを指定できる状態)で実行すること。

## ワークフロー

### Stage 0: PR情報の取得とルールの読み込み

1. 引数からPR番号またはURLを解釈する。引数がなければカレントブランチのPRを対象とする。

   ```bash
   gh pr view <番号> --json number,title,body,files,additions,deletions,labels,baseRefName
   gh pr diff <番号>
   ```

   diffが非常に大きい場合(数千行超)は全文を読まず、`--name-only` でファイル一覧を先に確認し、
   Stage 1 の判定に必要なパス情報を優先する。

2. `references/common/rules.md` を必ず読む。すべてのPRに適用される共通ルールが定義されている。

3. 変更ファイルとリポジトリ構成から対象スタックを検出し、対応する固有ルールがあれば追加で読む。
   固有ルールは `references/<スタック名>/rules.md` に置かれている。複数該当すればすべて読む。
   該当ディレクトリがなければ共通ルールのみで判定してよい。

   | 検出の目安 | スタック | 読むファイル |
   |---|---|---|
   | `*.unity` / `*.meta` / `Assets/` / `ProjectSettings/` | Unity | `references/unity/rules.md` |
   | `AndroidManifest.xml` / `build.gradle(.kts)` / `src/main/java\|kotlin` | Android | `references/android/rules.md` |

   新しいスタックのルールが追加されたら、この表にも行を足すこと(後述「新しいスタックへの対応」参照)。

### Stage 1: ルールによる即決(決定的判定)

読み込んだルール(共通 + 固有)を変更ファイル一覧・diffに適用する。

- **「人間レビュー必要」パターンに1つでも該当** → 判定を 🔴 で確定し、Stage 2 はスキップする。
  該当したルール名を判定根拠として記録する。危険領域の変更は内容の良し悪しにかかわらず
  人間の目を通すべきであり、LLMの評価で覆してはならない。
- **変更ファイルのすべてが「人間レビュー不要候補」パターンに該当** → diffをざっと確認し、
  パターンの想定どおりの変更内容(例: docsのみ、と見せかけたコード変更がない)であれば 🟢 で確定する。
  想定と異なる内容が混ざっていれば Stage 2 へ進む。
- **上記のどちらでもない** → Stage 2 へ進む。

### Stage 2: diffの精読によるリスク評価(グレーゾーンのみ)

diff全体を精読し、以下の5軸をそれぞれ 高 / 中 / 低 で評価する。
評価には必ず根拠(該当ファイル・該当箇所)を添える。根拠のない評価は判定の信頼を損なう。

| # | 評価軸 | 見るポイント |
|---|---|---|
| 1 | ロジックの複雑さ・挙動変更 | 条件分岐の追加変更、アルゴリズム変更、エッジケースの扱い |
| 2 | セキュリティ / データ整合性 | 入力検証、権限チェック、データの書き込み・削除経路 |
| 3 | 破壊的変更・後方互換性 | 公開インターフェースの変更、シリアライズ形式、設定値の意味変更 |
| 4 | テスト | 変更に対応するテストの有無・妥当性(テストがない挙動変更は危険信号) |
| 5 | 影響範囲 | 変更箇所の呼び出し元の多さ、共有コード・共通基盤かどうか |

判定ルール:

- 1つでも **高** がある → 🔴 人間レビュー必要
- すべて **低** → 🟢 人間レビュー不要
- それ以外(中が含まれる) → 🟡 人間レビュー推奨(優先度は中程度)

### Stage 3: レポート生成とPRコメント投稿

判定結果を以下のテンプレートで組み立てる(言語は日本語):

```markdown
## 🤖 レビュートリアージ判定: {🔴 人間レビュー必要 / 🟡 人間レビュー推奨 / 🟢 人間レビュー不要}

**判定方式**: {Stage 1: ルール即決 / Stage 2: diff精読}

### 判定根拠
{Stage 1 の場合: 該当したルールと対象ファイル}
{Stage 2 の場合: 5軸の評価表(軸 / 評価 / 根拠)}

### 要約
{変更内容とリスクの1〜3行の要約}

---
*このコメントは pr-review-triage スキルによる自動判定です。最終判断はチームのレビュー運用に従ってください。*
```

投稿の手順:

1. 組み立てたコメント全文をユーザーに提示し、**投稿してよいか必ず確認する**。
   PRコメントはチーム全員の目に触れる外部への出力であり、無断で投稿してはならない。
2. ユーザーが承認したら一時ファイル経由で投稿する:

   ```bash
   gh pr comment <番号> --body-file <一時ファイル>
   ```

3. ユーザーが投稿を望まない場合は、判定結果の表示のみで終了する。

## 新しいスタックへの対応

言語・フレームワーク・ツール固有のルールは `references/<スタック名>/rules.md` を追加するだけで拡張できる。

- ファイルは既存の `references/unity/rules.md` と同じ2表構成
  (「人間レビュー必要パターン」「人間レビュー不要候補パターン」)で書くこと。
  構成を揃えるのは、Stage 1 がどのスタックでも同じ手順でルールを適用できるようにするため。
- 追加したら、この SKILL.md の Stage 0 にあるスタック検出表に行を追加すること。
  検出表に載っていないルールファイルは読まれない。
