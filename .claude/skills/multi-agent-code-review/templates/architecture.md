## プロジェクト概要

<!-- アーキテクチャの前提となる情報を2〜3行で。採用しているスタイル
(レイヤードアーキテクチャ、Clean Architecture、MVVM 等)と、その目的。 -->

(例) 本プロジェクトは Clean Architecture を採用し、ドメインロジックをフレームワークから
分離している。

## レイヤー構成

<!-- レイヤーごとに、責務と配置パスを表で定義する。
レビューでは「変更ファイルのパス → 所属レイヤー」の判定に使われる。 -->

| レイヤー | 責務 | 配置パス |
|---|---|---|
| (例) Domain | ビジネスルール。フレームワーク非依存の純粋なコード | `src/domain/**` |
| (例) Application | ユースケース。Domain の組み合わせと調停 | `src/app/**` |
| (例) Infrastructure | DB・外部 API・フレームワークとの接続 | `src/infra/**` |
| (例) Presentation | UI・入出力 | `src/ui/**` |

## 依存方向ルール

<!-- 「どのレイヤーがどのレイヤーに依存してよいか」を明示する。
import / using / include の追加行がこのルールと照合される。 -->

### 許可される依存

- (例) Presentation → Application → Domain
- (例) Infrastructure → Domain(インターフェース実装のため)

### 禁止される依存

<!-- 違反が見つかると High(リスク high)として指摘される。 -->

- (例) Domain → 他のすべてのレイヤー(Domain は何にも依存しない)
- (例) Domain・Application → フレームワークの名前空間(`UnityEngine`、`androidx` 等)
- (例) Presentation → Infrastructure(必ず Application を経由する)

## 禁止事項

<!-- 依存方向以外の禁止パターン。見つかると High として指摘される。 -->

- (例) シングルトンの新規追加(既存の `ServiceLocator` を使う)
- (例) `src/domain/**` での I/O(ファイル、ネットワーク、DB)の直接実行

## 推奨パターン

<!-- 従うことが望ましいが、逸脱してもマージブロックしないもの(Medium 以下)。 -->

- (例) ユースケースは 1 クラス 1 公開メソッドとする
- (例) 外部 API のレスポンスは Infrastructure 層で Domain の型に変換してから渡す

## 例外

<!-- 上記ルールの適用外となる場所があれば書く(例: 実験用ディレクトリ、生成コード)。 -->

- (例) `src/experimental/**` は依存方向ルールの対象外
