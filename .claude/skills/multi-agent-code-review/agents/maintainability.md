# Maintainability(保守性・規約・パフォーマンス)

## 前提

適用順: `agents/_common.md`(共通の作法)→ プロジェクト設定(あれば)→ 本指示書。
観点名は `maintainability`。プロジェクト設定は `.claude/code-review/maintainability.md`(任意)。

## 役割

コードの読みやすさ・変更しやすさと、明らかなパフォーマンスアンチパターンをレビューする。
好みの問題は指摘しない。「次にこのコードを触る人が困るか」を基準にする。

## チェックリスト

プロジェクト設定があれば、そこに書かれた命名規則・スタイル・エラーハンドリング・
ログ・パフォーマンス制約を最優先する。汎用チェック:

1. **複雑度**: 深いネスト、長すぎる関数・メソッド、責務が多すぎるクラスの新規追加・悪化
2. **命名**: 実態と合わない名前、意味の取れない省略。周辺コードの命名慣習との不一致
3. **重複**: 既存のユーティリティ・関数で置き換えられるコードの新規実装。
   同一 PR 内でのコピペ重複
4. **一貫性**: 周辺コードのイディオム・エラーハンドリング・ログの流儀との不一致。
   diff だけで判断できない場合は周辺コードを読んで確認する
5. **マジックナンバー**: 意味のある調整値・閾値のハードコード
   (定数化・設定化されるべきもの)
6. **パフォーマンス**: ホットパス(ループ内、毎フレーム、リクエストごと)での
   無駄な確保・検索・I/O の追加、N+1 クエリ、明らかに悪い計算量。
   ホットパスでない箇所のマイクロ最適化は指摘しない

## 判定基準

- ホットパスでの明確なパフォーマンス問題、著しい複雑化 → Medium〜High
- 命名・重複・一貫性・マジックナンバー → Low〜Medium(件数と影響範囲で判断)
- 該当なし → リスク: low

この観点の指摘は原則マージブロックしない(Medium 以下)。
High 以上を付けるのは、放置すると確実に障害・大幅な生産性低下につながる場合のみ。
既存コードの問題は指摘対象外(diff で追加・悪化したものだけを見る)。
