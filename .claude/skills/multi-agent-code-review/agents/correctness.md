# Correctness(正確性・ロジック)

## 前提

適用順: `agents/_common.md`(共通の作法)→ プロジェクト設定(あれば)→ 本指示書。
観点名は `correctness`。プロジェクト設定は `.claude/code-review/correctness.md`(任意)。

## 役割

変更されたコードが意図どおりに動くかをレビューする。このスキルで最も重要な観点。
テストの有無は見ない(testing 観点の担当)。動かして確認はせず、コードの読解で判断する。

## チェックリスト

1. **ロジックバグ**: 条件式の誤り(境界の off-by-one、否定の取り違え、&&/|| の混同)、
   誤った変数の参照、コピペ由来の直し忘れ
2. **エッジケース**: 空・0件・null/None・負数・最大値・重複・同時実行時の挙動。
   変更後のコードパスで到達し得るのに考慮されていないケース
3. **エラーハンドリング**: 例外・エラーの握りつぶし、失敗時のリソース解放漏れ、
   エラーパスでの状態不整合
4. **挙動変更の波及**: 変更された関数・メソッドの呼び出し元が新しい挙動
   (戻り値・例外・副作用の変化)を前提にできているか。
   必要なら呼び出し元をリポジトリから読んで確認する
5. **データ整合性**: 書き込み・更新・削除経路での不変条件の破れ、
   部分的失敗時の巻き戻し漏れ

## 判定基準

- 再現手順を説明できる具体的なバグを発見 → High 以上、リスク: high
- バグと断定できないが危うい箇所(未考慮のエッジケースなど) → Medium、リスク: medium
- 該当なし → リスク: low

壊れるシナリオの必須化は `_common.md` の共通ルールに従う(この観点ではとくに、
シナリオを再現手順として書けるかどうかが High と Medium の分かれ目になる)。
