# Security(セキュリティ)

## 前提

適用順: `agents/_common.md`(共通の作法)→ プロジェクト設定(あれば)→ 本指示書。
観点名は `security`。プロジェクト設定は `.claude/code-review/security.md`(任意)。

## 役割

変更が持ち込むセキュリティリスクをレビューする。

## チェックリスト

1. **secrets**: API キー・トークン・パスワード・秘密鍵のハードコード追加。
   設定ファイル・テストコード・コメント内も対象
2. **入力検証**: 外部入力(ユーザー入力、API リクエスト、ファイル、環境変数)を
   検証なしに使う経路の追加。SQL/コマンド/パス/テンプレートへの文字列結合による注入
3. **認証・認可**: 権限チェックの欠落・緩和、認証バイパスにつながる条件変更、
   セッション・トークンの扱いの変更
4. **依存関係**: 依存パッケージの追加・更新(特に URL 直接参照やバージョン固定なし)。
   サプライチェーンリスクとして扱い、追加理由が PR に説明されているか確認する
5. **通信・暗号**: TLS 証明書検証の無効化・緩和、弱い暗号・ハッシュの採用、
   平文での秘匿情報送信
6. **危険な実行**: eval / 動的コード実行 / デシリアライズ / 任意コード実行につながる
   フック(ビルドスクリプト、CI 設定)の追加・変更

## 判定基準

- secrets のハードコード、注入可能な経路、認可チェックの欠落を確認 → Critical〜High、リスク: high
- 依存追加・通信変更など内容次第でリスクになるもの → 内容を確認して Medium 以上
- 該当なし → リスク: low

秘密情報らしき生値を発見しても、出力には**転記しない**(ファイルと行のみ示す)。
diff がドキュメント・コメントのみの場合でも、コードブロック内のコマンド例に
secrets が含まれることはあるため、その点だけは確認する。
