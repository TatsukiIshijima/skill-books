# 実装計画の作成

機能・タスクの要望を、実装作業に渡せる **実装計画** に落とし込む。進行は3段階で、それぞれ成果物を保存する:

1. **仕様作成** — 1問ずつ質問して仕様を確定し、仕様書を保存
2. **ADR 作成** — アーキテクチャ上重要な決定を ADR として記録
3. **計画書作成** — 仕様書・ADR を参照する実装計画を保存

位置づけ: **plan-feature(計画を作る)→ 実装・テスト → PR 作成**。導入先に実装用のスキルがあれば、生成した計画ファイルをそのまま渡せる。

---

## 各ドキュメントの役割

3つの成果物は役割を重複させない。**同じ内容を2つのドキュメントに書かない。他ドキュメントにある情報はパス参照で済ませる。** 実装セッションは計画書だけを最初に読めばよく、詳細が必要なときだけ仕様書・ADR を開く構造にする。

| ドキュメント | 役割 | 書くこと | 書かないこと |
|---|---|---|---|
| 仕様書 | **WHAT** — 要求の唯一の正 | 背景、スコープ、振る舞い、データの制約、受け入れ基準、未決事項 | 実装方式、ファイル構成、テスト手段 |
| ADR | **WHY** — 方式決定の記録 | 決定を迫った制約、採用方式と理由、代替案、トレードオフ | 要求の再説明(仕様書を参照)、実装手順 |
| 計画書 | **HOW** — 実装の道筋(実装者の入口) | 参照リンク、設計方針の結論、実装ステップ、テスト計画、完了条件、影響ファイル | 背景・スコープ・振る舞い・受け入れ基準の再掲(参照で済ます) |

---

## 引数

- `$ARGUMENTS` = 作りたい機能・タスクの概要。
- 空 → 直前の会話の文脈を対象にする。特定できなければユーザーに確認して停止。

## プロジェクト設定(任意)

導入先リポジトリの `.claude/plan-feature/config.md`(以降 `<config>`)があれば読み、以下に反映する。書式はこのスキルの `templates/config-example.md` を参照。

- 追加の質問観点(例: 特定のエンジン・フレームワーク・アーキテクチャに関する観点)
- 完了条件の品質ゲートの既定(例: TDD で進める、カバレッジ基準)
- 計画テンプレートへ追記するセクション・固定の留意点
- 保存先(既定: 仕様書 `docs/specs/` / ADR `docs/adr/` / 計画 `docs/plans/`)

`<config>` が無くても動作する。CLAUDE.md にある規約・方針は、`<config>` の有無にかかわらず常に設定に準ずるものとして反映する。

---

## 進め方の原則

- **相互理解に達するまで、体系的に質問して詰める。**
- **決定の依存関係を順に解決する**(前の決定が次の質問の前提になる順序で進める)。
- **質問は1問ずつ**提示する(`AskUserQuestion` を1問ずつ呼ぶ)。
- **各質問に推奨回答を添える**(推奨を先頭に置き「(推奨)」を付ける)。
- **コードベースや docs で答えられることは質問しない**。先に自分で調べる。

---

## 手順

### 1. 入力解決

- 上記ルールで対象の要望を確定する。曖昧・不在ならユーザーに確認して停止。
- 要望から短いスラッグを作る(以降の仕様書・ADR・計画のファイル名に使う)。

### 2. 事前調査(自分で調べる)

質問する前に、関係する範囲を読んで把握する。ここで分かることは質問に回さない。

- CLAUDE.md(ルート、および関連ディレクトリ配下にあるもの)
- `<config>`(あれば)
- README / `docs/` 配下の要件・仕様・設計ドキュメント(`docs/index.md` のような起点があればそこから辿る)。既存の仕様書(`docs/specs/`)・ADR(`docs/adr/`)があれば、関連するものを確認する
- 関連する既存コードと既存テストを Grep / Read で把握

### 3. 仕様の確定(1問ずつ、依存順に質問)

不明点を依存関係の順に1問ずつ詰める。各質問に推奨回答を添える。探索する観点(足りないものを順に):

- スコープ(対象 / 対象外)・受け入れ基準
- 実装の置き場所・責務分担(プロジェクトのアーキテクチャに沿って)
- 必要なデータ・設定値と、その持ち方(ハードコード / 設定ファイル / DB など)
- 外部との接点(UI・API・他モジュール・スキーマ)への影響
- エッジケース・異常系
- 既存コードへの影響範囲
- **テスト観点**: どのレベルのテストで何を保証するか(プロジェクトのテスト方針に沿って。TDD 採用プロジェクトなら先に書くテストのリスト)

`<config>` に追加観点があれば、同様に依存順へ組み込んで詰める。確定済み・自明な点は質問せず、仕様書・計画に直接書く。

質問の過程で **アーキテクチャ上重要な決定**(複数の実現方式から選択した、既存構造に影響する、後から変更しにくい)があったものを控えておく(ステップ5の ADR 候補)。

### 4. 仕様書の保存

**軽量パス判定**: 仕様確定が少数の質問(目安3問以内)で終わり、外部との接点や新規のデータ・設定値がほとんど無い小規模タスクでは、仕様書の作成を省略できる。省略する場合はユーザーに確認したうえで、確定した仕様を計画書の「仕様要約」セクションに数行で記載し、それを要求の正とする(このとき仕様書は作らないので役割分担の重複禁止には反しない)。ADR 候補の扱い(ステップ5)は変わらない。

確定した仕様を `docs/specs/<slug>.md`(`<config>` で指定があればそちら)に保存する(ディレクトリが無ければ作成)。既存の仕様書に書式の慣習があればそれに従う。無ければ次のテンプレートで日本語で記述する:

```markdown
# 仕様書: <タイトル>

## 概要・背景
<何を・なぜ>

## スコープ
- 対象: ...
- 対象外: ...

## 振る舞い仕様
- 正常系: ...
- 異常系・エッジケース: ...

## データ・設定値
<必要なデータ・パラメータとその制約。どう持つか(実装方式)はここに書かない>

## 外部との接点
<UI・API・他モジュールに対する振る舞いへの影響。無ければ「なし」>

## 受け入れ基準
- <満たすべき振る舞いを、検証可能な形で>

## 未決事項
<回答保留などで残った論点。無ければ「なし」>
```

### 5. ADR の作成

ステップ3で控えた ADR 候補ごとに、記録するかユーザーに確認してから作成する。候補が無ければこのステップはスキップし、その旨を報告する。

- 保存先: `docs/adr/NNNN-<slug>.md`(`<config>` で指定があればそちら)。既存の ADR の最大番号 +1 の4桁連番を振る(ディレクトリが無ければ `0001` から)。
- コンテキストには **決定を迫った制約のみを数行で** 書く。要求そのものの説明は仕様書のパス参照で済ませ、再掲しない。
- 既存の ADR にテンプレートや書式の慣習があればそれに従う。無ければ次の形式で日本語で記述する:

  ```markdown
  # NNNN. <決定のタイトル>

  日付: YYYY-MM-DD
  ステータス: 承認

  ## コンテキスト
  <決定を迫った課題・制約を数行で。要求の詳細は仕様書(docs/specs/<slug>.md)を参照>

  ## 決定
  <採用した方式と、その理由>

  ## 検討した代替案
  - <代替案>: <不採用の理由>

  ## 影響
  <この決定によって生じる利点・トレードオフ・追随が必要な箇所>

  ## 見直し条件
  <この決定を見直すべき観測可能な条件。例: 前提とした制約・規模・依存関係の変化。無ければ「なし」>
  ```

### 6. 計画の生成と保存

- `docs/plans/<slug>.md`(`<config>` で保存先指定があればそちら)に保存する(ディレクトリが無ければ作成)。
- 次のテンプレートで日本語で記述する。`<config>` に追記セクション・固定の留意点があれば反映する:

  ```markdown
  # 実装計画: <タイトル>

  <何をするかの1行要約。背景・要求の詳細は仕様書を参照>

  ## 関連ドキュメント
  - 仕様書: docs/specs/<slug>.md(要求・スコープ・振る舞い・受け入れ基準はこちら。本計画では再掲しない。軽量パスで省略した場合は「なし(本計画の仕様要約を正とする)」と書く)
  - ADR: docs/adr/NNNN-<slug>.md — <決定の1行要約>(無ければ「なし」)

  ## 仕様要約(軽量パスで仕様書を省略した場合のみ)
  - スコープ: ...
  - 振る舞い・受け入れ基準: ...

  ## 設計方針(結論のみ)
  - <採用した方式1>(経緯・代替案は ADR NNNN)
  - <採用した方式2>

  ## 実装ステップ
  1. ...
  (プロジェクトが TDD を採用している場合は Red → Green → Refactor の順で書き、ステップ単位で commit する旨を記す)

  ## テスト計画
  - [ ] <テストケース1: 期待する振る舞い>
  - [ ] <テストケース2>
  - ...

  ## 完了条件
  - 仕様書(軽量パスでは本計画の仕様要約)の受け入れ基準をすべて満たす
  - <品質ゲート(テスト方針・カバレッジ等)>

  ## 影響 / 新規ファイル
  - ...

  ## 留意点
  - 変更は本計画の範囲に限定する
  ```

- 完了条件の品質ゲートは、`<config>`・CLAUDE.md にテスト方針・品質基準があれば**必ずそれを反映**する。無ければ「関連テストがすべてグリーンであること」を最低限とし、質問で確認した基準を加える。振る舞いに関する基準は仕様書の受け入れ基準側に書き、ここには再掲しない。

### 7. 完了報告

- 保存した仕様書・ADR・計画ファイルのパスを提示する。
- 導入先に後続の実装スキルがあればその実行例を案内する。無ければ「この計画ファイルを渡して実装を依頼できます」と案内して終了。

---

## 停止・報告ポイント

- 対象の要望が特定できない → 確認して停止
- ユーザーが質問への回答を保留 → そこまでの内容で仕様書・計画を書くか確認(保留のまま書く場合は仕様書の「未決事項」に残す)

## 注意

- このスキルは **仕様書・ADR・計画の作成のみ**。実装・テスト・commit は行わない。
- 仕様書と ADR は永続的な記録なので、通常は commit する運用を想定している(commit は実装時・導入先の運用に従う)。計画ファイルを commit するか(保存先を gitignore するか)は導入先プロジェクトの運用に従う。
- 質問の進め方は grill-me(Matt Pocock 氏のスキル)の手法を参考にしている。
